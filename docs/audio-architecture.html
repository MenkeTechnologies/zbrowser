>_AUDIO ENGINE — IPC TOPOLOGY

The browser-wide always-on output DSP engine lives inside the sandboxed audio service (fork patches 0022/0024). The HUD Audio dashboard that drives it lives in a sandboxed renderer. Neither can reach the other directly — the config is a durable file, the meters an ephemeral file, and the unsandboxed browser process is the only party present on both sides of the sandbox.

The load-bearing fact

zwire-host and the audio service never exchange a message. zwire-host has no Chromium IPC (its whole surface is filesystem / OS / PTY / a pub-sub bus); the audio service is sandboxed and can open neither the file nor a socket. The file is the rendezvous; the browser process is the only party on both sides — filesystem on the zwire-host side, mojom on the audio side.

Because the config file is durable, it doubles as the persistence layer: bin/zwire reads it at launch and seeds the engine before any page or poller exists, so the DSP is shaped from the very first audio block.

Config — UI → engine

  1. 1audio.js buildSpec() → Native Messaging port, postMessage{cmd:'fs_write'} (audio.js:201)
  2. 2zwire-host writes the spec to $STATE/audio-eq
  3. 3browser RepeatingTimer 25ms → ReadFileToString (0024:88,128) — the sandbox blocks the audio process from reading it
  4. 4browser → mojom AudioService::SetZwireEqConfig → ZwireAudioEq atomically swaps the process-global config (0024:107)

Meters — engine → UI

  1. 5audio service OnMoreData() → ZwireMeterSnapshotJson (Goertzel spectrum + peak/RMS + correlation + scope) (0022)
  2. 6browser mojom ZwireMeters.Pull 33ms on a ThreadPool sequence → writes $STATE/meters (0024:169,199)
  3. 7zwire-host fs_watch → pushes {ev:'meter'} frames (audio.js:838)
  4. 8audio.js onMessage parses JSON into the charts — zero page-side polling (audio.js:839)
